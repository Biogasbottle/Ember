# PRD: V2.1 Triple Barrier + Fractional Diff + Multi-Horizon

## Problem Statement

V2's binary direction label (`direction_24h`) is essentially a coin flip — 24H forward direction is near-random. 20 features are all lagging indicators with no anticipatory signal. Financial time series non-stationarity is ignored (return distributions differ dramatically between bull/bear regimes). Single-horizon prediction cannot resolve conflicting signals across timeframes.

## Solution

Three concurrent improvements:

1. **Triple Barrier Labeling**: Replace binary direction with triple-barrier labels (upper/lower price barriers + time horizon). Filters out low-magnitude noise samples and encodes both direction AND conviction in a single label.
2. **Fractional Differentiation**: Use fractional differencing (d=0.3) instead of integer returns, preserving long-memory structure while achieving approximate stationarity.
3. **Multi-Horizon Consensus**: Train separate models for 4H/12H/24H horizons. Only output predictions when all three models agree, filtering out conflicting signals.

## User Stories

1. As a researcher, I want Triple Barrier labels to replace binary direction, so that low-magnitude noise samples are filtered out
2. As a researcher, I want fractional differentiation to preserve long-memory structure, so that features are stationary without losing trend information
3. As a researcher, I want 3-horizon consensus prediction, so that only high-confidence signals are used
4. As a researcher, I want a reusable `feature_lab/labels.py` module, so that label generation logic is centralized and testable
5. As a researcher, I want fractionally differenced feature variants in the registry, so that the feature engineering pipeline generates stationary input columns
6. As a researcher, I want `build_features.py` to compute labels for all 3 horizons simultaneously, so that training can use consistent data
7. As a researcher, I want the training script to evaluate consensus accuracy alongside individual model metrics, so that I can measure signal improvement

## Implementation Decisions

### Triple Barrier Labeling

- Horizontal barriers: `±2 * rolling_std(return, 60)` from entry price
- Vertical barrier (time): 6 bars (24H), 3 bars (12H), 1 bar (4H)
- Label values: `+1` (touches upper first), `-1` (touches lower first), `0` (expires untouched)
- New module: `feature_lab/labels.py` with `generate_triple_barrier_labels(df, horizon_bars, vol_multiplier)` function
- Called by `build_features.py` to produce `label_4h`, `label_12h`, `label_24h` columns

### Fractional Differentiation

- Order: `d = 0.3` — preserves long memory, achieves approximate stationarity
- Applied to: `close`, `oi`, `funding`, `volume` raw columns
- Produces helper columns: `_frac_close_d03`, `_frac_oi_d03`, etc.
- Feature expressions in registry reference these frac-diff columns
- Implementation: fixed-window weighted sum in `build_features.py` (no external library needed)

### Multi-Horizon Consensus

- Three separate LightGBM Classifier models, each trained on its own label (label_4h / label_12h / label_24h)
- At prediction time: `consensus = majority_vote(model_4h.proba, model_12h.proba, model_24h.proba)`
- Only consensus-positive samples counted in final metrics; non-consensus marked `uncertain=-1`
- Consensus ratio logged: what fraction of test samples achieve 3-way agreement

## Testing Decisions

- CF test: `build_features.py` output parquet contains `label_4h`, `label_12h`, `label_24h` as integer columns (-1/0/1)
- CF test: `train_lightgbm.py` trains 3 models, produces consensus metrics, all model files exist
- Unit test: `labels.py` — Triple Barrier with all-up market (all +1), all-down (all -1), sideways (all 0)
- Unit test: `labels.py` — fractional diff output same length as input, first values are finite
- All tests follow existing pattern in `tests/test_pipeline.py`

## Out of Scope

- Walk-Forward backtest (V4)
- Liquidation features
- Market regime detection / HMM
- Additional data sources (on-chain, order book)

## Further Notes

- Existing 20 features are retained; some are supplemented with frac-diff variants
- The `build_features.py` entry point signature remains unchanged
- The `train_lightgbm.py` entry point signature gains an optional `horizon` parameter for per-model training
- Triple Barrier label distribution should be monitored: if >70% of labels are 0, the barrier width needs tuning
